# Protobuf Contracts

Protobuf is **optional** in this architecture. Use it when you need
network transport or multi-language clients. When all callers are
in-process, plain Go contract DTOs are simpler.

## When to Use Protobuf

**Use protobuf when:**
- Services communicate over the network (Connect/gRPC)
- You need schema evolution with backward compatibility
- You want generated clients in TypeScript, Kotlin, etc.
- You need `buf lint` to enforce API design rules

**Skip protobuf when:**
- All callers run in the same process (use contract DTOs)
- Prototyping or early development

## Transport Swapping

The key advantage of combining Connect RPC with the contract pattern:
the same inbound adapter serves both in-process and network callers.

**In-process (default in the monolith):**

```
cmd/mmw/main.go
  └── deffoomod.NewInprocClient(fooModule)
        └── calls fooModule methods directly (no network, no serialisation)
```

**Over-the-network (when splitting to microservices):**

```
barmod process                     foomod process
  └── connect.NewFooServiceClient     └── Connect HTTP server
        └── HTTP/2 + protobuf               └── FooHandler (same code)
```

The `FooHandler` Connect implementation changes nothing. Only the
composition root swaps `deffoomod.NewInprocClient(fooModule)` for a
`connect.NewFooServiceClient(httpClient, url)`.

## Hybrid Approach

Start with Go DTO contracts. Add protobuf when you need the network:

**Phase 1 — Go DTOs only (contract definition)**

```go
// contracts/definitions/foomod/dto.go
type CreateFooRequest struct { Title, Priority, OwnerID string }
type FooResponse      struct { ID, Title, Status, Priority, OwnerID string }
```

**Phase 2 — Protobuf DTOs (network-ready)**

```go
// contracts/definitions/foomod/dto.go
import foov1 "github.com/example/mmw-contracts/gen/go/foo/v1"

// Use generated types as the contract DTOs directly
type CreateFooRequest = foov1.CreateFooRequest
type FooResponse      = foov1.FooResponse
```

Migrate when you need it, not before.

## Connect vs gRPC-Gateway

This architecture uses [`connectrpc.com/connect`](https://connectrpc.com),
not the gRPC-Gateway pattern. Connect differences:

| | Connect | gRPC-Gateway |
|---|---|---|
| Transport | HTTP/1.1 + HTTP/2 | HTTP/2 only (gRPC) |
| Browser clients | Native (no proxy) | Requires Envoy proxy |
| Generated code | `connect-go` plugin | `grpc-gateway` plugin |
| Interceptors | `connect.Interceptor` | gRPC middleware |
| buf plugin | `buf.build/connectrpc/go` | `buf.build/grpc-ecosystem/gateway` |

## Connect Handler Structure

The generated interface (from `buf generate`) that the inbound adapter implements:

```go
// Generated by buf: contracts/gen/go/foo/v1/foov1connect/foo.connect.go
package foov1connect

type FooServiceHandler interface {
    CreateFoo(context.Context, *connect.Request[foov1.CreateFooRequest]) (*connect.Response[foov1.CreateFooResponse], error)
    GetFoo(context.Context, *connect.Request[foov1.GetFooRequest]) (*connect.Response[foov1.GetFooResponse], error)
    CompleteFoo(context.Context, *connect.Request[foov1.CompleteFooRequest]) (*connect.Response[foov1.CompleteFooResponse], error)
    DeleteFoo(context.Context, *connect.Request[foov1.DeleteFooRequest]) (*connect.Response[foov1.DeleteFooResponse], error)
    ListFoos(context.Context, *connect.Request[foov1.ListFoosRequest]) (*connect.Response[foov1.ListFoosResponse], error)
}

func NewFooServiceHandler(svc FooServiceHandler, opts ...connect.HandlerOption) (string, http.Handler)
```

The adapter implements this interface and is registered via `NewFooServiceHandler`.
